## 開場：Token Optimization Tech Stack

跨層降低 LLM token 成本的槓桿（Cross-layer levers）：

- **L5 產品／使用者層** — Pricing model、Tier、Usage cap、SLA（user-facing price 定義，不在本課範圍內）
- **L4 應用層 Application（Main optimization stage）**
  - 1 GENERATION（壓 token 數量）：caveman（output soft compression ~3×）、structured-CoT（thinking hard compression / grammar 22–43×）、reasoning effort（reasoning budget control，knob）
  - 2 CONTEXT（管 context 重用）：Semantic cache（vector similarity hit ~70%）、Context compaction（SDK auto summarize ~60%）、RAG vs long context（retrieval trade-off 1,250×）
  - 3 ARCHITECTURE（系統設計選擇）：LLM routing（inter-model dispatch 10–60×）、Multi-agent（cost explosion risk +15×）、MCP overhead（tool / protocol overhead 5–134K）
- **L3 API 定價層**：TIME（Batch −50%++ / Priority +150%）、MEMORY（Prompt cache read −90%）、CAPABILITY（model tier price gap 5–51×）、COMMITMENT（PTU / CCU flat-rate guarantee）、PREDICTION（Predicted outputs）、FORMAT（Structured outputs / function call）、CUSTOMIZATION（Fine-tune / Distillation / LoRA）
  - Hidden gotchas：tokenizer drift、cache invalidation
- **L2 推論計算層 Inference Compute**：vLLM、SGLang、llama.cpp｜PagedAttention、Continuous batching、Quantization
- **L1 訓練計算層 Training Compute**：H100 / B200 / TPU、PyTorch / JAX｜$10M–$1B+ CAPEX
- **L0 訓練資料層 Training Data（Emerging 2026）**：PROTOCOL（HTTP 402 / x402 / Web Bot Auth / llms.txt）、PLATFORM（Cloudflare Pay Per Crawl / AI Crawl Control）、DEALS（NYT $300M+ / Reddit $60M/yr）、1B+ HTTP 402/day on Cloudflare
- **Cross-layer**：Observability（Langfuse / Helicone / LangSmith）｜Self-hosting（Ollama / vLLM / LiteLLM）

## 你的 Agent 管線，模型怎麼配製？（AgentOpt）

Columbia 大學 DAPLab 的 **AgentOpt** 論文，用 9 個模型、81 種組合的大規模實驗，顛覆了「最強的模型放在最重要的位置，就會得到最好的結果」這個假設。

### 用 Opus 當 Planner？

同一任務、同一 Agent 管線架構，只調換 Planner / Solver：

| 組合 | 準確率 |
|---|---|
| 🏆 Ministral 8B（Planner）+ Opus（Solver） | 74.27% |
| ❌ Opus（Planner）+ 任何模型（Solver） | 31.71% |

- 差距 **42 個百分點**，而贏的那組 Planner 價格是 Opus 的 **1/33**
- Ministral 8B 為 Mistral AI 8B 端側小模型，定價 $0.15/百萬 token

### 最反直覺的實驗結果（HotpotQA 多跳問答）

- Ministral 8B（Planner）+ Claude Opus 4.6（Solver）= 74.27% 🏆
- Claude Opus 4.6（Planner）+ 任何 Solver = 31–32%（倒數 11 名全是 Opus 當 Planner）

### 為什麼最強模型反而最差？

- Opus 太強，強到不願意委派任務
- 論文分析執行日誌：9 種 Opus 當 Planner 的組合中，有 **7 種** 出現「不委派」行為

### 跨基準的系統性發現

| # | 基準 | 最佳組合與結論 |
|---|---|---|
| 1 | HotpotQA（多跳問答） | Ministral 8B + Opus = 74.27%；Planner 需要的不是「聰明」而是「守規矩」 |
| 2 | MathQA（數學推理） | Opus（Answerer）+ Haiku 4.5（Critic）= 98.84%；9 個 Critic 準確率差距僅 2.9 pp |
| 3 | GPQA Diamond（研究生科學問答） | 唯一「用最強模型就對了」；Opus 單獨 = 74.75% |
| 4 | BFCL v3（多輪函數呼叫） | 三模型並列 70%，成本差 32×；同分選便宜的 |

### 「直接用最好的模型」到底多浪費？（成本倍數差距）

| 基準 | 成本倍數差距 |
|---|---|
| HotpotQA | 21× |
| MathQA | 118× |
| BFCL | 32× |

- MathQA 最極端：Opus+Opus 與 Ministral+Haiku 準確率相近，但成本差 **118×**

### 為什麼不能靠直覺選模型組合？

- 測 8 種搜索演算法；LM Proposal（GPT-4.1 推薦組合）結果令人沮喪：
  - GPQA：表現尚可（直覺有效）
  - HotpotQA：僅 34%（遠低於最佳 74.27%）
  - BFCL：僅 45%
- Hill Climbing 在 HotpotQA 上只有 52% 機率找到最佳組合（局部搜索走不到全局最優）
- 🏆 贏家 **Arm Elimination**：基於信賴區間逐輪淘汰，節省 24–67% 搜索預算
- 核心：Pipeline 角色交互是 cross-stage，benchmark 表現好 ≠ pipeline 角色表現好

### Anthropic Advisor Tool

| | 做法 |
|---|---|
| 過去 | Opus 當主腦（Planner），便宜模型當手腳（Executor） |
| Advisor Tool | Haiku / Sonnet 跑主迴圈當執行者，Opus 退居幕後當「顧問」，只在關鍵時刻被呼叫 |

- 據 TestingCatalog：Haiku + Opus 顧問大幅超過 Haiku 單獨運行，總成本仍低於直接用 Sonnet
- 容錯設計：顧問呼叫失敗，執行者不中止（很多時候不需顧問也能搞定）

### 任務類型決定最佳分工

- **需要工具使用**（如 HotpotQA）：弱模型當 Planner 可能更好（不搶活幹，乖乖委派）
- **純推理**（如 GPQA）：強模型獨挑
- **需要批判審查**（如 MathQA）：Critic 模型選擇影響不大，不值得花大錢
- **多步驟長程**（如 SWE-bench）：便宜模型跑主迴圈 + 強模型按需諮詢

### 你現在可以做什麼？

1. 審計 Agent 管線成本（列出每角色模型與每月 token；注意「假高價值」角色）
2. 警惕「能力過剩」陷阱（太聰明繞過該遵守的流程）
3. 系統化找最佳組合（AgentOpt 開源，支援 LangGraph / AutoGen / CrewAI；Arm Elimination 省 40–60% 搜索預算）
4. 別相信直覺，包括 AI 的直覺（GPT-4.1 在 HotpotQA 推薦只有 34%）

### 一句話總結

- Agent 管線優化的單位不是「單一模型的能力」，而是「模型組合在特定任務上的匹配度」
- 用錯組合，可能花 32× 的錢，拿到一半的效果

### Q&A 重點

- 小模型不是永遠適合當 Planner（GPQA 上 Opus 獨挑最佳；沒有普適最佳組合）
- 2 角色 pipeline：9 模型 × 2 角色 = 81 組合，暴力搜索約 $50–120，Arm Elimination 降到 $30–70；月 API > $500 即值得
- 與 LLM Router 差異：Router 針對「單一查詢」路由（省單次呼叫），AgentOpt 針對「整條 pipeline」做組合優化（省整條管線）

## Cost Stack 框架總覽

### 為什麼需要 Cost Stack 框架？（三種誤解）

- ❌ 誤解一：Token 經濟學就是用 prompt 叫模型講話精簡一點
- ❌ 誤解二：省 token 就是 prompt engineering 的副產品
- ❌ 誤解三：Cache 跟 batch 就是全部的解法
- ✅ 正解：Token 成本來自 6 個不同層次，每層有截然不同的 trade-off 邏輯與工程代價

### 主圖：Cost Stack（12 個機制 × 6 個成本層）

- Money flows top-down，optimization happens at **L4**
- 越往下的層次，工程師的干預空間越大，但也越需要嚴謹的 eval 支撐

### 12 案例對應到層級

| 層 | 名稱 | 重點 |
|---|---|---|
| L5 | 產品／使用者層 | 定價、freemium、用量上限、SLA（不在本課範圍，但要記住存在） |
| L4 ★主場 | 應用層 Application | 工程師 90% 的優化機會在這｜GENERATION / CONTEXT / ARCHITECTURE |
| L3 | API 定價層 | Provider 提供的 7 大契約機制（每個都是一個 trade） |
| L2 | 推論計算層 | speculative decoding（2–3×）、PagedAttention、continuous batching（多數對 app dev 不可見） |
| L1 | 訓練計算層 | $10M–$1B+ capital expenditure（對 app dev 完全不可見） |
| L0 ★NEW 2026 | 訓練資料層 | Cloudflare Pay Per Crawl、x402、內容授權 = Token 經濟學新上游 |

- 跨層：Self-hosting（跨 L1–L3，自營模型由你支付）｜LLM FinOps / Observability（跨所有層，量測你在哪一層花錢）

## §1 L0 層 — 訓練資料層：Cloudflare Pay Per Crawl

### L0 的核心命題

- 前 11 個案例都在處理 token「已被產生之後」的事；Cloudflare 把上游從不可觀測變成可觀測，首次在訓練資料流引入價格訊號

| 數字 | 意義 |
|---|---|
| 1B | 每日 HTTP 402 回應（全球爬蟲請求規模） |
| $60M | Reddit ↔ OpenAI 實際成交年費 |
| 73,000 | 爬蟲：人類比（Anthropic 爬蟲 vs 真實訪客） |

- 過去：爬蟲索引 → 搜尋 → 點擊，流量循環存在
- 現在：AI 直接消費內容，循環消失；Token 經濟學第一次有上游價格訊號

### HTTP 402 = SDD 奇蹟示範

- 時間線：1996 標記 "Reserved for future use" → 2025 AI 爬蟲經濟成熟，402 被啟用（等待整整 30 年）
- 學生最常的抗議：「現在不知道未來需求，寫 spec 沒用。」
- 402 的反問：30 年前 spec 設計者也不知道 AI 爬蟲會出現 — 但他們留了空間。好的 spec 是 forward-compatible 的，這正是 SDD 框架的核心命題

### 機制與依賴（建立在 Verifiability 之上）

- 沒有 cryptographic bot identity 就無法計費
- 流程：AI 爬蟲請求 → WAF + Bot 身分驗證 → 規則判定或 402 → 付款處理與交付

## §2 L3 層 — API 定價層：Provider 7 大契約機制

### 7 大軸 — 每個都是一個 Trade

| 軸 | 你給出 | 你拿到 |
|---|---|---|
| 時間 | 容忍延遲 | −50% / +150% |
| 記憶 | Prefix 穩定 | Cache read −90% |
| 能力 | 接受較低能力模型 | 5–51× 價差 |
| 承諾 | 預付 / 包月 | Flat rate |
| 預測 | 預測一部分輸出 | 該段免費 |
| 格式 | 接受結構化輸出 | 間接省 retry |
| 客製 | Fine-tune 投資 | 長期單價降 |

### 時間軸：Batch / Flex / Standard / Priority（OpenAI）

| Tier | 折扣 / 加價 | 延遲承諾 |
|---|---|---|
| Batch | −50% | 最多 24 小時 |
| Flex | 約 −50% | 即時但時段變動 |
| Standard | 0%（基準） | 即時 |
| Priority | +150%（2.5×） | 比 Standard 更快 |

- Priority 是反向 trade — 用金錢換速度，適合 latency 極敏感的即時互動

### 記憶軸：Prompt Caching

| Provider | Cache 寫入 | Cache 讀取 |
|---|---|---|
| Anthropic | 1.25×（5 min）/ 2×（1 hr） | 0.1×（−90%） |
| OpenAI | 無額外費用 | 0.5–0.1× |
| Google | 有寫入費用 | 大幅折扣 |

- Break-even：Anthropic 5-min cache 讀 1 次就划算；1-hour cache 讀 2 次以上才划算
- ⚠ System prompt / tool schema 改動 → invalidate 全部 cache（tool 定義位於 cache hierarchy 最前端）

### 能力軸：Model Tier

| Provider | 同家族價差範圍 | 倍數 |
|---|---|---|
| Anthropic | Haiku 4.5 ($1/$5) → Opus 4.7 ($5/$25) | 5–25× |
| OpenAI | GPT-5 nano → GPT-5.5 | 51× |
| Google | Flash-Lite → Pro | ~10× |

- 隱性陷阱：Opus 4.7 新 tokenizer 對同樣文字多產生 35% tokens；每 token 單價沒漲，但 effective cost 已漲 35%，不能只看標價

| 文字 | 舊 tokenizer (Opus 4.6) | 新 tokenizer (Opus 4.7) |
|---|---|---|
| Hello world | 2 tokens | 2 tokens |
| 你好世界 | 4 tokens | 5 tokens |
| 我們正在討論成本經濟學 | ~9 tokens | ~12 tokens |
| def fibonacci(n): | 6 tokens | 7–8 tokens |

### 其他 4 個契約機制

1. **Predicted Outputs**（OpenAI 獨有）：猜對的部分免費，用例 code editing；需 spec 完整到包含預期 output
2. **Structured Outputs**：用 special tokens 表達結構，避免 retry 浪費；function calling 屬此類
3. **Provisioned Throughput**：commit-based，從 per-call 升維到容量規劃；適合流量穩定可預測
4. **Fine-tune / Distillation**：長期單價降，但需 stable workload 與完整 eval pipeline，初期投入高

### Case Study：Predicted Outputs（深入）

- 起點：500 行 code editing 場景，LLM 從第一個 token 重新生成整個函數，90% 內容相同卻全付費
- 痛點：output tokens 比 input 貴 3–5×（autoregressive 逐 token）；code editing 約 90% output 只是抄輸入；inline suggestion 對 latency 極敏感
- 機制（反向操作）：client-side speculative decoding — 客戶端提供預測內容，inference 時並行驗證，命中即免費，不命中走標準計費

API 用法：`messages` 外多一個 `prediction` 欄位（`{"type": "content", "content": original_code}`）

| Token 類別 | 計費 |
|---|---|
| Input tokens | 正常收費 |
| Accepted prediction tokens（猜對） | FREE ✅ |
| Rejected prediction tokens（猜錯） | 付 output 費 ⚠️ |

- 關鍵 caveat：猜錯也付錢，accuracy 直接決定 ROI（90% 猜對大省 / 50% 沒省多少 / 30% 反而更貴）
- OpenAI 官方：code refactoring 3–5× faster，最多省 50%+ output cost

### DEEPER INSIGHT：預測軸 — 你用「已知資訊」換折扣（SDD 邏輯極限）

- 七軸中預測軸獨特：其他軸交換「限制」，預測軸交換「你已經擁有的資訊」
- SDD 命題邏輯極限：spec 越完整 → 成本越低；spec 完整到包含預期 output 本身 → 成本降到接近 0（LLM 只在做 verification 不在 generation）
- 三個工程提醒：
  1. 量 acceptance rate（production 必追蹤）
  2. 不適合 reasoning model（o-series、Claude thinking 均不支援，推理過程不可預測）
  3. 與 cache 互動複雜（同用 Prompt Caching 須小心 invalidate 規則）
- 收尾要點：核心機制 = client-side speculative decoding；最佳場景 = code refactoring / inline suggestion；關鍵指標 = acceptance rate

## §3 L4-GENERATION 層：caveman / structured-cot / reasoning effort

### L4-G 三案例光譜

| 機制 | 控制方 | 約束類型 | 壓縮倍數 |
|---|---|---|---|
| caveman | App dev | SKILL.md（軟） | ~3× |
| structured-cot | App dev | Grammar（硬） | 22–43× |
| reasoning effort | Provider | Enum knob | ~2–5× |

- 核心洞察：spec formality 與壓縮倍數成正比（從軟性指引到硬性 grammar 是連續 trade-off 曲線）

### Case Study：caveman

- 哲學：「why use many token when few do trick」— 用 SKILL.md 軟性指引產生極簡輸出
- 強制度光譜：正常 → caveman lite → caveman full → caveman ultra → wenyan-full（文言文）
- 數字：output token 縮減 65%（benchmark 平均）；整體壓縮 ~3×（軟性約束上限）

### Case Study：structured-cot

- 20 行 GBNF grammar 強制思考結構，達到無訓練、無 fine-tune、無 reward model 的推理壓縮
- 機制：inference time logit masking — 每個 token 生成步驟，grammar 直接遮蔽不合法 token，零訓練成本

grammar 範例：
```
root  ::= think code
think ::= "\n" "GOAL: " line "APPROACH: " line "EDGE: " line "\n\n"
line  ::= [^\n]+ "\n"
code  ::= [\x09\x0A\x0D\x20-\x7E]+
```

三組對照（揭示 formal vs informal spec 的 enforcement gap）：

| Mode | Mean Think Tokens | 相對壓縮 |
|---|---|---|
| FREE | 3,087 | 1× |
| PROMPT_TERSE（自然語言請求簡潔） | 2,298 | 1.34×（沒用） |
| FSM（grammar） | 138 | 22.4× |

- 結論：prompt 請模型簡潔 → 答應了沒做到；grammar 強制 → 額外 16.65× 壓縮（formal spec 執行力優勢）

暗面：Reasoning Displacement

| 指標 | FREE | FSM_PLAN |
|---|---|---|
| Mean think tokens | 52 | 1,753（+33×） |

- 看 token 壓縮 43.3×，看 total tokens 壓縮只有 5.0×
- Verifiability Mindset 教科書：作者主動找會打臉自己的 metric 並誠實呈現（推理被擠到 code comments 裡，表面亮麗、真實壓縮有限）

### Case Study：Reasoning Effort

| Provider | 機制 | 預設 | 可選值 |
|---|---|---|---|
| OpenAI | o-series / GPT-5 `reasoning.effort` | medium | minimal / low / medium / high / xhigh |
| Anthropic | Claude 4.6+ `thinking:{"type":"adaptive","effort":...}` | 須顯式啟用 | low / medium / high |
| Google | Gemini `reasoning_effort` | 視 model | enum |
| DeepSeek V4 | 內建調節，唯一顯示真實 thinking tokens | — | — |

- 普遍問題：很多開發者啟用 extended thinking 後從不調 effort，為不需推理的請求付 thinking tokens

量化結果（OpenAI，medium → minimal）：

| 任務類型 | Thinking tokens 縮減 | Latency 改善 | Accuracy 影響 |
|---|---|---|---|
| 簡單 factual Q&A | −80% | −3× | 幾乎無 |
| 工具呼叫 | −60% | −2× | −2 pp |
| 複雜推理 | −40% | −1.8× | −15 pp ⚠️ |

### structured-cot vs Reasoning Effort 對偶（SDD 完美 ablation）

| | structured-cot | Reasoning Effort |
|---|---|---|
| 控制方 | App developer | Provider |
| 約束 | Hard（grammar） | Soft（enum） |
| 需要 | Open-weights + llama.cpp | API + reasoning model |
| 壓縮倍數 | 22–43× | ~2–5× |
| Spec 形式 | Formal grammar | Discrete enum |

- 互補：API 場景用 Reasoning Effort 快速降成本，Open-weights 場景用 Grammar 追求極限壓縮

## §4 L4-CONTEXT 層：cache / compaction / RAG / MCP

### Case Study：Semantic Caching

- Prompt cache 抓「同一 prefix 重複」；Semantic cache 抓「不同問法但意思相同」

| 數字 | 來源 |
|---|---|
| API calls −68.8% | GPT Semantic Cache (arXiv) |
| Hit rate 67% | VentureBeat 真實案例 |
| Cost −73% | VentureBeat 真實案例 |
| Hit rate 12× 提升 | VectorQ vs static threshold |

- 主流工具：GPTCache、Bifrost、VectorQ
- Verifiability 風險流程：查詢嵌入 → 向量 DB 相似度（cosine，門檻 0.8）→ ≥0.8 回 cached，否則呼叫 LLM

### Case Study：Context Compaction

- Anthropic first-party feature，API 參數觸發自動摘要過長 context：
  `context_management={"edits":[{"type":"compact_20260112"}]}`

| 數字 | 說明 |
|---|---|
| −58.6% tokens | Customer service workflow（122K → 86K） |
| 60–80% | 一般 context 縮減（大多數對話場景） |
| ~95% 觸發率 | Claude Code 預設（保留 ~33K buffer） |

失敗模式：

- ⚠ Compaction 會 summarize away CLAUDE.md 的 critical rules，導致模型忘記重要約束
- Workaround：critical rules 寫進 system prompt（不在 conversation history，不被 compaction 影響）；自訂 compactPrompt 明確保留檔案路徑、錯誤訊息、使用者糾正
- Proactive context engineering 是工程，reactive /compact 是急救

### Case Study：RAG vs Long Context 經濟學

| 指標 | RAG | Long Context（1M） |
|---|---|---|
| Per-query cost | $0.00008 | $2.00 |
| 成本倍數 | 1× | 25,000× |
| Latency | 1× | 30–60× 慢 |
| Mid-doc recall | 高 | ~60%（lost in the middle） |

- Elasticsearch Labs：RAG 便宜 1,250× per query
- 注意：Anthropic 近期 Sonnet/Opus 4.6 提供 flat-rate 1M context（無 surcharge），長 context 成本曲線快速變動，需持續追蹤

### 三條 Context 壓縮路徑彼此正交

- caveman / structured-cot → 攻擊 Output / Thinking tokens
- Context Compaction → 攻擊 Conversation context
- RAG → 攻擊 Knowledge context
- 三條可疊加（理論乘法收益），但每加一層 eval 負擔也疊加

### Case Study：MCP / Tool Use Hidden Cost

| 數字 | 說明 |
|---|---|
| 55K | 5 個 MCP server = 58 tools，光 schema 就佔 55,000 tokens |
| 17K | Jira MCP 單獨 token overhead |
| 134K | 極端案例（優化前 overhead，對話尚未開始） |

- 對話還沒開始，context window 已被 tool schema 消耗殆盡（multi-agent 最隱性成本）

對比：Traditional vs Code Execution（同一任務處理 1,000 筆，差 166×）

| 方法 | token | 說明 |
|---|---|---|
| ❌ Traditional | 200,000 | 1,000 records × 200 tokens/筆，逐筆傳送 |
| ✅ Code Execution | 1,200 | MCP Code Execution（subprocess + AST），壓縮 166× |

- 場景：Salesforce 撈 Q4 deals → 篩選 → 整理 → gmail → slack，第 1 步回傳 1,000 筆拖垮 agent loop
- 為什麼省：資料留在 sandbox 不進 context；filter/aggregate 在 code 端做；多 tool composability；可程式化中間邏輯（for loop、條件判斷不應交給 LLM）

Anthropic 三個解法：

1. **Token-efficient tool use beta**（Claude 4 預設）：output tokens 平均 −14%，最多 −70%
2. **Tool Search Tool**：按需發現工具，不再 eager-load 所有 schema
3. **Programmatic Tool Calling**：用 code execution 處理 intermediate results（如 166× 案例）

### Skill vs MCP Agent — 架構選擇 = Token Cost 選擇

| 維度 | Skill | MCP |
|---|---|---|
| Load 時機 | Lazy（用到才讀 SKILL.md） | Eager（連接就 load schema） |
| Token cost | 低 | 高 |
| 適合 | 確定要用的能力 | 探索性、多元能力組合 |

- 加 MCP server 像加 npm package：以為免費，但每加一個吃 5–17K tokens；架構決策在第一行程式碼之前就決定了成本結構

## §5 L4-ARCHITECTURE 層：routing / multi-agent / self-hosting

### Case Study：LLM Routing

- 不壓縮 token，但讓每 token 變便宜 10–60×（不同難度任務分配不同能力模型）

AWS 真實案例（每天 50,000 歷史題 + 50,000 數學題）：

| | 月成本 |
|---|---|
| 全部 Claude 3.5 Sonnet（無路由） | $14,850 |
| 路由分流後 | $8,044（節省 46%） |
| Routing overhead | $108–189（僅佔節省金額 1.3%） |

三種主要 Routing 策略：

| 策略 | Accuracy | Latency | Cost | 說明 |
|---|---|---|---|---|
| Static（hard-code） | 高 | 最低 | 最低 | 規則固定 |
| LLM-assisted | 高 | ~0.5s | 中 | 用 LLM 判斷 |
| Semantic | 中 | ~0.1s | 極低 | Embedding 相似度 |
| Hybrid | 高 | 中 | 中 | 混合策略 |

- 學術系統 LLMRouter（ulab-uiuc）：16+ 演算法統一 benchmark
- Router 失敗很安靜（仍回答，只是貴或答得差）→ 必須量測 misroute rate

### Case Study：Multi-Agent Cost Explosion ⚠️

| 倍數 | 場景 |
|---|---|
| 1× | 一般 Chat（基準線） |
| 4× | Single Agent + Tools |
| 7× | Claude Code Agent Teams |
| 15× | Multi-Agent System |

- Token usage 解釋 80% performance variance

Anthropic 官方兩段警語：

- Multi-agent systems are best suited for tasks where result value far exceeds cost.
- Domains that require all agents to share the same context or involve many dependencies between agents are not a good fit for multi-agent systems today.

適合 vs 不適合：

- ✅ 適合：breadth-first research、independent parallel directions、result value 高（例：Claude Research、大型 codebase 平行探索）
- ❌ 不適合：需要 shared context、agent 間 dependencies 多、線性任務（例：大部分 chat 應用）
- 學生愛拆 sub-agent 因為「比較專業」→ 15× cost 應讓設計者重新思考 result value 是否撐得起代價

### Case Study：Self-Hosting Break-Even

| 來源 | Break-even 條件 | 月費用 |
|---|---|---|
| TokenMix | 月 API spend $20K | — |
| digitalapplied (code) | 600M tokens/月 | — |
| digitalapplied (chat) | 1.2B tokens/月 | — |
| Vucense (RTX 4090) | 硬體 8 個月回本 | $200/月 |

- 2026 open-weight 已追上 frontier 水準，自建財務邏輯成立
- 隱性殺手：Engineer time = $750–3,000/月（人力成本常超過硬體本身）

Self-Hosting 工具光譜：

- **Ollama**：本地開發、個人實驗首選，部署簡單
- **vLLM**：Production 首選，PagedAttention + continuous batching
- **TensorRT-LLM**：NVIDIA 極致效能，高吞吐量生產環境
- **SGLang**：MoE 友好，結構化生成支援佳；NCKU lab 建議混合架構（self-host 處理預測 bulk，Claude API 處理 high-value bursty）

## §6 跨層：LLM FinOps / Observability — 讓優化可被驗證

### 為什麼 Observability 是 Token 經濟學的前提

- Anthropic / OpenAI dashboard 只告訴你「多少 tokens」，不告訴你「為什麼」（哪個功能、用戶、session）
- 沒有 observability → 前面 12 個優化都是盲目操作，無法 verify 是否生效、無法診斷退化根源

### 工具光譜

- **Langfuse**：MIT 授權、open-source、self-hostable，目前最廣泛使用
- **Helicone**：proxy-based，一行程式碼接入，最快上手
- **LangSmith**：與 LangChain/LangGraph 生態深度整合
- **LangWatch**：LLM-native eval 能力強
- **CloudZero / Finout**：enterprise 等級，跨 provider 成本分配
- **Datadog LLM Monitoring**：已有 Datadog 基建的團隊最快擴展

### 必量測的指標（Verifiability Mindset 營運層 instance）

1. Cost per feature / user / session（最核心成本歸因）
2. P50 / P95 Latency（長尾延遲反映隱性 retry / fallback）
3. Cache Hit Rate（分類：prompt vs semantic，各有 quality 風險）
4. Tool Definition Overhead（每 request tool schema token，早期發現 MCP 膨脹）
5. Reasoning Token Consumption（per request thinking token，檢查過度思考）
6. Error / Retry Rate（agent loop 失控最早期信號）

## §7 機制疊加：乘法 ROI

### 機制可疊加 — 真實計算示範

| 步驟 | 機制 | 該層縮減 | 累積有效成本 |
|---|---|---|---|
| 0 | Baseline | — | 1.00× |
| 1 | Batch | × 0.5 | 0.50× |
| 2 | Prompt Cache（80% prefix） | × 0.28 | 0.14× |
| 3 | Routing Opus → Sonnet | × 0.6 | 0.084× |
| 4 | caveman 砍 50% output | × 0.7 | ~0.06× |
| 5 | structured-cot 砍 thinking | × 0.4 | ~0.024× |

- 理論可達 ~40× 整體成本縮減（$1,000/月 → $25/月）

### 但每多一層機制…

- 工程複雜度上升（更難除錯）
- Eval 負擔上升（每層需獨立 eval pipeline，不能共用）
- Failure mode 數量上升（機制間互相干擾，組合失敗）
- 真正試煉不是「能不能省」，是「能不能 verify 沒搞壞東西」

### ROI 排序建議

1. **Prompt Caching** — 最低投入、立即見效，讀一次即回本
2. **Batch** + 3. **Routing** — Batch 適合非即時 workload；Routing 在異質性高時 ROI 最大
4. **caveman** + 5. **Semantic Cache** — output 壓縮與重複 query 攔截，需基礎 eval
6.–8. 進階機制 — structured-cot、Context compaction / RAG 改造，需更成熟 eval 能力
8. **Self-Hosting** — 最大投入、最大潛在收益，需穩定 workload 與工程團隊

## §8 框架對應：對應 AIASE 五大框架

12 個機制是五大框架的具體 instance：

- **Harness Engineering**：structured-cot、Routing、Self-hosting（工程師如何駕馭模型能力邊界）
- **SDD（Spec-Driven Development）**：Provider 機制、structured-cot、HTTP 402、Reasoning effort（spec 形式決定行為可預測性）

- **Eval-Driven Development**：structured-cot 三組對照、FinOps、Multi-agent eval（用數據驅動決策，主動找反例）
- **Skill vs Sub-Agent / A2A**：Multi-agent cost、MCP overhead、Routing、Agent-as-customer（架構選擇即成本選擇）

## 收尾：課程總結

### 一句話總結

- **契約設計**：Token 經濟學的本質，是在每一層設計一份清醒的契約
- **每個機制都是一個 Trade**：你讓出某些條件換取成本優勢；沒有免費的壓縮，只有清醒的交換
- **工程能力 = 看穿代價**：在每一層清楚知道你在交換什麼，並選擇能承受的代價組合
- **Verifiability 是底線**：無法量測的優化不算優化；每個決策都需要 eval 支撐

### 第 12 個案例的特殊地位

- 前 11 個案例教你「怎麼花得更少」（Generation / Context / Architecture 各層壓縮優化）
- 第 12 個（Cloudflare L0）教你「這個錢是怎麼產生的、誰在收」（從訓練資料上游理解 token 生命週期）
- L0 是整個 cost stack 的起點，也是最常被忽視的一層
